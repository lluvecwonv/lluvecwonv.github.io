## 개요

텍스트-투-SQL(Text-to-SQL) 생성 문제는 자연언어 질문을 SQL 쿼리로 변환하는 작업으로, 데이터베이스 인터페이스의 접근성을 크게 향상시킬 수 있다. 최근 대규모 언어모델(Large Language Models, LLM)의 급속한 발전으로 이 분야에도 혁신적인 변화가 나타났다. 본 논문은 LLM을 활용한 텍스트-투-SQL 생성 기법들을 종합적으로 분석하는 서베이이다.

이 논문은 텍스트-투-SQL 생성 방법론을 네 가지 주요 카테고리로 분류하고, 각 방식의 장단점, 적용 가능한 시나리오, 그리고 성능 차이를 체계적으로 비교한다. 또한 벤치마크 데이터셋, 평가 지표, 그리고 현재 남아있는 주요 과제들을 상세히 논의한다.

## 1. 서론 (Introduction)

### 텍스트-투-SQL의 중요성

데이터베이스는 현대 정보 기술의 핵심 인프라이지만, SQL 쿼리 작성에는 전문적인 지식이 필요하다. 텍스트-투-SQL 생성은 사용자가 자연언어로 의도를 표현하면 이를 자동으로 실행 가능한 SQL 쿼리로 변환하는 기술이다. 이는 데이터베이스에 대한 접근성을 크게 낮출 수 있어 매우 중요한 연구 주제이다.

### LLM의 등장과 전환점

ChatGPT, GPT-4, Claude 등 대규모 언어모델의 등장은 텍스트-투-SQL 분야에 획기적인 변화를 가져왔다. 이전의 기법들은 대부분 특정 데이터셋에 특화된 모델을 학습하는 방식이었지만, LLM은 다양한 데이터셋에 대해 강력한 생성 능력을 보여준다. 뛰어난 자연언어 이해 능력과 코드 생성 능력으로 인해 LLM은 텍스트-투-SQL 생성에 최적화된 도구가 될 수 있다.

### 본 서베이의 목표

본 논문의 목표는 LLM을 활용한 텍스트-투-SQL 생성 기법의 최신 동향을 종합적으로 정리하고, 각 방법론의 특징과 성능을 비교하며, 미래의 연구 방향을 제시하는 것이다.

## 2. 텍스트-투-SQL 생성 방법론의 분류

본 서베이에서는 LLM을 활용한 텍스트-투-SQL 생성 방법론을 다음과 같이 4가지 주요 카테고리로 분류한다.

### A. 프롬프트 엔지니어링 (Prompt Engineering)

프롬프트 엔지니어링은 모델의 가중치를 수정하지 않고, 입력 프롬프트를 설계하여 LLM의 성능을 향상시키는 방식이다.

#### A1. Zero-Shot 프롬프팅
- **개념**: 별도의 예제나 학습 없이, 지시사항만으로 작업 수행
- **장점**: 신속한 배포, 추가 학습 비용 불필요
- **단점**: 복잡한 쿼리나 도메인별 특이성에 약함
- **적용 사례**: 기본적인 SQL 생성, 빠른 프로토타이핑

#### A2. Few-Shot 프롬프팅
- **개념**: 프롬프트에 소수의 예제(typically 2-5개)를 포함하여 작업 학습
- **장점**: Zero-shot보다 정확도 향상, 추가 학습 비용 없음
- **단점**: 예제 선택에 따라 성능 변동 가능
- **전략**:
  - 구도 기반 선택 (구조적으로 유사한 예제)
  - 의미론 기반 선택 (의미적으로 유사한 예제)
  - 무작위 선택 (베이스라인)

#### A3. Chain-of-Thought (CoT) 프롬프팅
- **개념**: 중간 추론 과정을 명시적으로 표현하도록 유도
- **절차**: "먼저...를 찾고, 다음으로...를 한 후, 마지막으로 SQL을 작성한다"
- **장점**: 복잡한 쿼리에서 정확도 향상, 해석 가능성 증대
- **예제**:
  ```
  데이터베이스 스키마를 검토한다.
  자연언어 질문의 의도를 파악한다.
  필요한 테이블과 컬럼을 확인한다.
  SQL 쿼리를 단계적으로 구성한다.
  ```

#### A4. Plan-and-Solve (PaS) 프롬프팅
- **개념**: 쿼리 생성 전에 체계적인 계획 수립
- **계획 단계**:
  - 질문 분석
  - 데이터베이스 스키마 검토
  - JOIN 전략 수립
  - WHERE 조건 정의
  - SELECT 항목 결정
- **장점**: 대규모 스키마와 복잡한 쿼리에서 효과적
- **단점**: 계획 생성 추가 시간 소요

#### A5. Uncertainty-Guided 프롬프팅
- **개념**: 모델의 불확실성을 감지하고, 불확실한 부분에 대해 추가 정보나 다른 전략 제공
- **절차**:
  1. 초기 SQL 생성
  2. 신뢰도 점수 계산
  3. 신뢰도 낮은 부분에 대해 재생성 또는 다른 프롬프트 적용
- **장점**: 동적 적응, 오류 자동 교정
- **적용**: 특히 복잡한 SQL이나 드물게 나타나는 쿼리 타입

### B. 파인튜닝 (Fine-Tuning)

파인튜닝은 사전 학습된 LLM의 가중치를 특정 데이터셋과 작업에 맞게 조정하는 방식이다.

#### B1. 전체 파라미터 파인튜닝 (Full-Parameter Fine-tuning)
- **개념**: 모델의 모든 파라미터를 학습 가능하도록 업데이트
- **특징**:
  - 높은 계산 비용
  - 대규모 GPU 메모리 필요
  - 최고의 성능 향상 가능
- **적용**:
  - 충분한 계산 자원이 있을 때
  - 최고 성능이 필수적일 때
  - 도메인별 특화가 중요할 때
- **결과**: 특정 데이터셋에서 70-90% 정확도 달성 가능

#### B2. 파라미터 효율적 파인튜닝 (Parameter-Efficient Fine-tuning, PEFT)

##### LoRA (Low-Rank Adaptation)
- **원리**:
  ```
  원래 파라미터 W를 고정하고,
  낮은 랭크의 행렬 쌍 (A, B)을 추가
  W' = W + αBA^T (α는 스케일 계수)
  ```
- **장점**:
  - 1-5%의 파라미터만 학습
  - 메모리 사용 대폭 감소 (60-70%)
  - 빠른 학습
  - 모듈식 적용 가능
- **성능**: 전체 파인튜닝의 95-99% 수준
- **적용 예**:
  - 소규모 팀이나 제한된 자원 환경
  - 여러 도메인에 대한 빠른 적응
  - 프로덕션 배포

##### QLoRA (Quantized LoRA)
- **개념**: LoRA에 양자화(Quantization)를 결합
- **특징**:
  - 4-bit 정밀도 사용
  - 메모리 사용 75% 이상 감소
  - 싱글 GPU에서 대규모 모델 파인튜닝 가능
- **트레이드오프**: 약간의 성능 감소 (1-3%)

### C. 태스크별 특화 학습 (Task-Specific Training)

특정 텍스트-투-SQL 작업에 최적화된 아키텍처와 학습 전략.

#### C1. Mixture of Experts (MoE) 아키텍처
- **개념**: 여러 개의 전문가 모듈이 입력에 따라 활성화
- **구조**:
  ```
  입력 → 라우터 → 선택된 전문가들 → 출력
  ```
- **전문가 설계**:
  - 스키마 이해 전문가
  - JOIN 처리 전문가
  - WHERE 조건 생성 전문가
  - ORDER BY/GROUP BY 전문가
  - 집계 함수 전문가
- **장점**: 각 부분별 특화, 높은 효율성
- **단점**: 복잡한 구조, 학습 난이도 높음

#### C2. Transformer 기반 아키텍처
- **특징**:
  - 표준 인코더-디코더 구조 활용
  - 스키마 인코딩 방식 개선
  - 컬럼/테이블 값 임베딩 포함
- **개선 사항**:
  - 스키마 관계 그래프 추가
  - 의미론적 유사성 기반 주의 메커니즘
  - 계층적 인코딩 (테이블 → 컬럼 → 값)
- **성능**: 기본 Transformer 대비 15-25% 향상

#### C3. CNN 기반 아키텍처
- **개념**: 합성곱 신경망을 텍스트-투-SQL에 적용
- **적용 방식**:
  - 1D 합성곱을 시퀀스 인코딩에 사용
  - 다중 필터 크기로 다양한 패턴 학습
  - 풀링으로 중요 특성 추출
- **장점**: 병렬 계산 효율성, 빠른 추론
- **단점**: 장거리 의존성 학습에 약함 (Transformer 대비)

### D. 에이전트 기반 시스템 (Agent-Based Systems)

LLM을 에이전트로 활용하여 반복적으로 SQL을 개선하는 방식.

#### D1. 다중 에이전트 협업 (Multi-Agent Collaborative)
- **구조**:
  ```
  쿼리 분석 에이전트 → 스키마 이해 에이전트 → SQL 생성 에이전트
  → 검증 에이전트 → (필요시) 개선 에이전트
  ```
- **각 에이전트의 역할**:
  - **쿼리 분석**: 자연언어 질문 분해, 의도 파악
  - **스키마 이해**: 관련 테이블/컬럼 선택, 관계 파악
  - **SQL 생성**: 선택된 정보로 SQL 작성
  - **검증**: SQL 문법 검증, 스키마 일치성 확인
  - **개선**: 오류 수정, 성능 최적화
- **장점**: 각 단계별 전문성, 투명성, 오류 추적 용이
- **단점**: 오버헤드, 네트워크 호출 증가

#### D2. 반복 개선 (Iterative Refinement)
- **프로세스**:
  1. 초기 SQL 생성
  2. 실행 결과 피드백
  3. 오류 또는 부정확성 감지
  4. 프롬프트 수정 및 재생성
  5. 반복 (최대 n번)
- **개선 전략**:
  - 문법 오류 수정
  - 스키마 불일치 해결
  - 로직 오류 교정
  - 효율성 최적화
- **성능**: 초기 생성 대비 10-30% 정확도 향상 가능
- **비용**: 반복 횟수에 따라 토큰 사용량 증가

#### D3. 동적 피드백 루프 (Dynamic Feedback Loops)
- **개념**: 데이터베이스 실행 결과를 피드백으로 활용
- **피드백 종류**:
  - **실행 오류**: 문법 오류, 테이블/컬럼 미존재
  - **의미론적 오류**: 결과는 나왔지만 질문과 맞지 않음
  - **성능 오류**: 쿼리가 너무 느림
- **루프 구조**:
  ```
  자연언어 → SQL 생성 → 데이터베이스 실행
  → 결과 평가 → (오류 있으면) 재생성 → 반복
  ```
- **검증 방식**:
  - 자동 검증 (SQL 문법, 스키마)
  - 휴리스틱 검증 (예상 결과 범위)
  - 예제 기반 검증 (유사 질문의 결과와 비교)

#### D4. 도구 통합 (Tool Integration)
- **외부 도구**:
  - **SQL 검증 도구**: SQL 문법 검사, 최적화 제안
  - **스키마 탐색 도구**: 데이터베이스 구조 쿼리, 샘플 데이터 조회
  - **실행 엔진**: SQL 직접 실행, 결과 반환
  - **의미론적 검색 도구**: 유사한 과거 쿼리 검색
  - **테이블/컬럼 문서화**: 메타데이터, 설명, 사용 예제
- **통합 방식**:
  - API 호출
  - 함수 정의
  - 플러그인 시스템
- **이점**: 정확성 향상, 실시간 정보 반영, 오류 감소

## 3. 벤치마크 데이터셋 (Benchmark Datasets)

### 단일 도메인 데이터셋 (Single-Domain Datasets)

#### ATIS (Airline Travel Information System)
- **규모**: 약 5,000개의 자연언어-SQL 쌍
- **도메인**: 항공사 예약 시스템
- **특징**:
  - 상대적으로 작은 규모
  - 간단한 스키마 (약 8개 테이블)
  - 비교적 단순한 쿼리
- **사용 목적**: 초기 방법론 검증, 기준 비교

#### GeoQuery
- **규모**: 약 900개의 자연언어-SQL 쌍
- **도메인**: 지리 정보 데이터베이스
- **특징**:
  - 매우 작은 규모
  - 지리적 관계 쿼리 포함
  - 공간 함수 사용
- **난이도**: 상대적으로 낮음

#### Scholar
- **규모**: 약 15,000개의 자연언어-SQL 쌍
- **도메인**: 학술 논문 검색 시스템
- **특징**:
  - 중간 규모 스키마 (약 15개 테이블)
  - 복잡한 JOIN 쿼리
  - 의미론적 복잡성 높음
- **난이도**: 중간 ~ 상

#### Advising
- **규모**: 약 4,570개의 자연언어-SQL 쌍
- **도메인**: 대학 학생 상담 시스템
- **특징**:
  - 도메인 특화적 언어
  - 관계형 데이터 처리
  - 다양한 속성 쿼리

### 교차 도메인 데이터셋 (Cross-Domain Datasets)

#### Spider
- **규모**: 약 11,000개의 자연언어-SQL 쌍
- **도메인 수**: 200개의 데이터베이스, 138개 도메인
- **특징**:
  - 가장 광범위한 도메인 커버리지
  - 복잡한 스키마 (평균 테이블 수: 5-20개)
  - 다양한 쿼리 타입:
    - 중첩 쿼리 (Nested queries)
    - 복합 JOIN (Multiple JOINs)
    - 집계 함수 (Aggregation)
    - GROUP BY / ORDER BY
    - UNION 쿼리
- **어려움**: 도메인 변화에 따른 일반화
- **평가**: 정확한 매치(EM), 실행 정확도(EX) 포함

#### WikiSQL
- **규모**: 약 80,000개의 자연언어-SQL 쌍
- **도메인 수**: 다양한 Wikipedia 테이블 (약 26,000개)
- **특징**:
  - 대규모 데이터셋
  - 단순한 쿼리 (SELECT, WHERE, ORDER BY만)
  - 단일 테이블 쿼리
  - 높은 어휘 다양성
- **난이도**: Spider보다 낮음
- **용도**: 대규모 학습 가능

### 증강/견고성 변형 데이터셋 (Augmented/Robustness Variants)

#### 데이터 증강 기법
- **자동 생성**:
  - 유사 질문 생성 (질문 의도는 같으나 표현 다름)
  - 동등 SQL 생성 (다른 SQL이지만 결과는 같음)
  - 패러프레이징 (자연언어 다양화)

#### 견고성 테스트 셋
- **OOD (Out-of-Distribution) 테스트**:
  - 학습 데이터에 없는 도메인
  - 새로운 스키마 구조
  - 드문 쿼리 타입

- **적대적 예제**:
  - 의도적으로 모호한 질문
  - 함정이 있는 질문
  - 미묘한 차이가 있는 질문

## 4. 평가 지표 (Evaluation Metrics)

| 지표 | 설명 | 계산 방식 | 범위 |
|------|------|---------|------|
| **Exact Match (EM)** | 생성된 SQL이 참값 SQL과 정확히 일치하는 비율 | 정확히 일치한 샘플 수 / 전체 샘플 수 | 0-100% |
| **Execution Accuracy (EX)** | SQL을 데이터베이스에서 실행한 결과가 참값과 일치하는 비율 | 올바른 결과를 낸 샘플 수 / 전체 샘플 수 | 0-100% |
| **Valid Efficiency Score** | 생성된 SQL의 문법적 유효성과 효율성을 평가 | 유효한 쿼리만 계산, 실행 시간 고려 | 0-100% |
| **Test-Suite Accuracy** | 여러 테스트 케이스에서 SQL이 올바른 결과를 산출하는 비율 | 모든 테스트 케이스 통과 / 전체 테스트 케이스 | 0-100% |
| **근사 일치 (Partial Match)** | SQL 구조의 일부가 일치하는 비율 (부분 신용) | 일치한 구성 요소 수 / 전체 구성 요소 수 | 0-100% |

### 지표 선택 기준

- **EM**: 정확성을 엄격히 평가할 때 사용
- **EX**: 실제 데이터베이스 작동 관점에서 평가할 때 사용
- **Valid Efficiency**: 실시간 시스템이나 성능이 중요한 경우
- **Test-Suite**: 복잡한 쿼리나 엣지 케이스 처리 능력 평가

  - 스키마 문서화 (있는 경우 활용)

#### 자연언어 정제
- **토큰화**: 언어별 형태소 분석
- **정규화**: 대소문자 통일, 특수문자 처리

- **구도 기반 검색**:

  ```
  1. 쿼리 구조 추출 (SELECT, WHERE, JOIN 복잡도)

  ```
- **의미론적 유사도 기반**:
  - 질문과 참값 SQL의 의미론적 유사도 계산
  - BERT 기반 임베딩 사용
  - 상위 K개 선택

#### 프롬프트 템플릿

```
[Few-Shot 예제들]
```

### 학습 설정

#### 하이퍼파라미터

| 파라미터 | 값 | 설명 |
|---------|-----|------|
| Learning Rate | 5e-5 | 기본 학습률 |
| Batch Size | 32 | 그래디언트 축적 크기 |
| Epochs | 3-5 | 에포크 수 |

#### 데이터 분할
- **학습/검증/테스트**: 70/15/15 또는 80/10/10

- **계층화 샘플링**: 쿼리 복잡도별 균형

### 생성 설정

#### 빔 서치 (Beam Search)
- **Beam Width**: 5-10
- **최대 길이**: SQL 최대 토큰 수 (100-200)
- **최소 길이**: 약 10 토큰

#### 온도 및 Sampling
- **Temperature**:
  - Zero-shot: 0.1-0.3 (결정적)
  - Few-shot: 0.3-0.5 (다양성)
- **Top-P Sampling**: 0.9-0.95
- **Top-K**: 40-50

### 평가 절차

#### 정확도 평가
1. **문법 검증**: SQL 파서로 문법 오류 검사
2. **스키마 검증**: 테이블/컬럼 존재 여부 확인
3. **실행 검증**:
   - 데이터베이스 실행 시도
   - 오류 메시지 수집
4. **결과 검증**:
   - 쿼리 결과가 예상 결과와 일치하는지 확인
   - 정확한 행/값 비교

#### 통계 분석
- **신뢰 구간**: 95% 신뢰도로 계산
- **유의성 검정**: t-검정 또는 Mann-Whitney U 검정
- **오류 분석**: 실패 사례의 패턴 분류

## 7. 남아있는 주요 과제 및 한계

### 1. 자연언어 모호성 (Natural Language Ambiguity)

#### 문제점
- 동일한 자연언어가 여러 SQL로 해석 가능
- 사용자 의도의 암묵적 가정
- 도메인 특화 표현의 이해

#### 예시
```
질문: "최근 매출을 조회하세요"
가능한 해석:
1. 어제의 매출
2. 지난 주의 매출
3. 지난 달의 매출
4. 지난 분기의 매출
```

#### 현재 접근법의 한계
- Few-shot 예제도 모든 경우를 다룰 수 없음
- 컨텍스트 정보 부족
- 사용자 의도 확인 메커니즘 부재

### 2. 복잡한 스키마 처리 (Complex Schema Understanding)

#### 문제점
- 테이블 수 증가에 따른 성능 저하
- 숨겨진 관계 이해 어려움
- 동의어 테이블/컬럼명 처리

#### 스키마 크기별 성능 저하

| 테이블 수 | EM 성능 | 성능 저하 |
|----------|---------|---------|
| 2-5개 | 85-90% | - |
| 5-10개 | 75-82% | 10% |
| 10-20개 | 65-75% | 20% |
| 20개 이상 | 55-70% | 30% |

#### 해결 시도
- 관련 테이블만 선택하는 필터링
- 계층적 스키마 인코딩
- 스키마 그래프 표현

### 3. 정교한 쿼리 생성 (Sophisticated Query Generation)

#### 어려운 쿼리 타입
- **중첩 쿼리**: 3 단계 이상의 중첩 → 성능 50% 이상 저하
- **다중 JOIN**: 4개 이상의 테이블 JOIN → 복잡도 지수적 증가
- **창 함수 (Window Functions)**: 현재 LLM이 적절히 생성하지 못함
- **재귀적 쿼리 (CTE)**: 일반적 텍스트-투-SQL 데이터셋에 거의 없음
- **고급 함수**: 도메인별 특화 함수 (지리함수, JSON 함수 등)

#### 성능 분석

| 쿼리 타입 | 성능 | 난이도 |
|----------|------|--------|
| SELECT WHERE | 90%+ | 매우 낮음 |
| Single JOIN | 80-85% | 낮음 |
| Multiple JOIN (2-3) | 70-75% | 중간 |
| Multiple JOIN (4+) | 50-65% | 높음 |
| 중첩 쿼리 (1 level) | 70-75% | 중간 |
| 중첩 쿼리 (2+ levels) | 45-60% | 매우 높음 |
| 창 함수 | 40-55% | 매우 높음 |
| CTE 재귀 | <30% | 거의 불가능 |

### 4. 도메인 적응 (Domain Adaptation)

#### 문제
- 학습 도메인과 다른 도메인에서 성능 급격히 저하
- 도메인별 특화 어휘와 관례
- 비즈니스 로직 암묵적 가정

#### Zero-Shot 성능 분석 (Spider 도메인별)

| 도메인 | EM | EX | 기타 |
|--------|----|----|------|
| 평균 | 71% | 78% | - |
| 최고 (금융) | 85% | 90% | 데이터 정규화 특징 |
| 최저 (헬스케어) | 55% | 62% | 의료 용어 복잡함 |
| 편차 | 30% | 28% | 도메인 간 큰 차이 |

### 5. 설명 가능성 및 투명성 (Explainability & Transparency)

#### 문제점
- 왜 이 SQL을 생성했는지 설명 불가
- 오류의 원인 파악 어려움
- 사용자 신뢰도 낮음

#### 현재 접근법
- CoT: 중간 추론 과정 표시
- 에이전트: 단계별 의사결정 로깅
- 한계: 여전히 불충분함

- **목표**: 더 다양하고 현실적인 벤치마크
- **방향**:
  - 산업별 데이터셋 (금융, 의료, 전자상거래)
  - 실시간 데이터 변화 반영
  - 보안/개인정보 보호 고려
  - 다국어 데이터셋 확대

#### 7. 효율성 개선
- **목표**: 추론 비용/시간 감소
- **방향**:
  - 모델 경량화 (Distillation)
  - 캐싱 및 재사용 (유사 쿼리)
  - 배치 처리 최적화
  - 로컬 모델과 API 모델의 하이브리드

### 최종 평가

LLM을 활용한 텍스트-투-SQL 생성은 이미 실제 응용에 충분한 수준에 도달했다. ChatGPT-4와 같은 최신 모델들은 85% 이상의 정확도를 달성하고 있으며, 에이전트 기반 시스템을 통해 더욱 향상된 성능을 기대할 수 있다.

다만 다음의 과제들이 해결되어야 한다:
- 자연언어의 모호성 처리
- 복잡한 스키마 이해
- 정교한 쿼리 생성 능력
- 도메인 간 일반화

이러한 과제들은 기술의 지속적 발전과 함께, 인간-AI 협업 인터페이스의 발전을 통해 점진적으로 해결될 것으로 예상된다.

## 리뷰 노트

이 서베이 논문은 LLM을 활용한 텍스트-투-SQL 생성의 현황을 매우 체계적으로 정리한 우수한 자료이다.

**강점**:
1. 4가지 명확한 카테고리로의 분류가 이해하기 용이함
2. 각 방법론별 성능 비교가 정량적으로 제시됨
3. 실험 설정이 매우 상세하게 설명됨
4. 남아있는 과제와 한계를 명확히 제시함
5. 향후 연구 방향이 구체적으로 제안됨

**약점**:
1. 각 기법별 상세한 구현 방식은 제한적
2. 계산 비용 대비 성능 분석이 더 깊이 있을 수 있음
3. 상용 API(GPT-4 등)에 대한 과도한 의존성 언급 부족
4. 보안 및 개인정보 보호 관점의 논의 없음

**종합 평가**: 이 서베이는 텍스트-투-SQL 생성 분야의 현황을 파악하려는 연구자들과 개발자들에게 매우 가치 있는 참고 자료이다. LLM 시대의 텍스트-투-SQL 생성에 대한 포괄적이고 심도 있는 이해를 제공한다.
