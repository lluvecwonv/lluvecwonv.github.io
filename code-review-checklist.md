# 코드 리뷰 체크리스트 (Code Review Checklist)

> 버전 1.0 | 작성일: 2026-03-31
> 이 체크리스트는 팀의 기술 스택과 상황에 맞게 커스터마이징하여 사용하세요.

---

## 🧭 사용 방법

- **PR 작성자**: 제출 전 셀프 체크
- **리뷰어**: 리뷰 시 참고 기준
- ✅ 통과 | ⚠️ 논의 필요 | ❌ 수정 필요

---

## 1. 정확성 (Correctness)

### 기능 요구사항
- [ ] PR 설명과 실제 변경 내용이 일치한다
- [ ] 구현이 티켓/이슈의 요구사항을 완전히 충족한다
- [ ] 엣지 케이스(빈 값, null, 경계값)가 처리되어 있다
- [ ] 비즈니스 로직에 오류가 없다
- [ ] 기존 기능이 깨지지 않는다 (regression)

### 에러 처리
- [ ] 예외 상황이 적절히 처리되어 있다
- [ ] 에러 메시지가 사용자 또는 개발자에게 의미 있다
- [ ] 실패 시 롤백 또는 복구 로직이 있다 (필요한 경우)
- [ ] 서드파티 API 장애 상황도 고려되어 있다

### 데이터 정합성
- [ ] DB 트랜잭션이 올바르게 처리된다
- [ ] 데이터 타입이 명확하고 일관성 있다
- [ ] 날짜/시간 처리 시 타임존이 고려되어 있다

---

## 2. 보안 (Security)

### 입력값 검증
- [ ] 모든 사용자 입력에 대해 서버 사이드 유효성 검사가 있다
- [ ] SQL Injection 방지 (파라미터화된 쿼리 또는 ORM 사용)
- [ ] XSS 방지 (출력 시 적절한 이스케이프)
- [ ] 파일 업로드 시 확장자/크기 검증이 있다

### 인증 & 인가
- [ ] 인증이 필요한 엔드포인트에 인증 미들웨어가 적용되어 있다
- [ ] 권한(Role) 체크가 올바르게 구현되어 있다
- [ ] JWT/세션 처리가 안전하다
- [ ] 다른 사용자의 데이터에 접근할 수 없다 (IDOR 방지)

### 민감 정보
- [ ] 비밀번호, API Key, 토큰이 코드에 하드코딩되어 있지 않다
- [ ] 환경 변수 또는 Secret Manager를 사용한다
- [ ] 로그에 민감 정보(PII, 카드번호 등)가 출력되지 않는다
- [ ] `.gitignore`에 민감 파일이 포함되어 있다

### 통신 보안
- [ ] 외부 통신은 HTTPS를 사용한다
- [ ] CORS 설정이 과도하게 허용적이지 않다

---

## 3. 성능 (Performance)

### 쿼리 & 데이터베이스
- [ ] N+1 쿼리 문제가 없다 (필요 시 JOIN 또는 batch 처리)
- [ ] 자주 조회되는 컬럼에 인덱스가 있다
- [ ] 불필요하게 많은 데이터를 한 번에 불러오지 않는다 (pagination 고려)
- [ ] 무거운 쿼리는 캐싱을 고려한다

### 코드 효율성
- [ ] 불필요한 반복 연산이 없다
- [ ] 루프 내에서 I/O 작업(DB, API 호출)을 피한다
- [ ] 메모리 누수 가능성이 없다 (이벤트 리스너 해제 등)
- [ ] 비동기 처리가 필요한 곳에 적용되어 있다

### 리소스 관리
- [ ] 파일/커넥션 등 리소스가 사용 후 적절히 해제된다
- [ ] 대용량 파일/데이터는 스트리밍 처리를 고려한다

---

## 4. 유지보수성 (Maintainability)

### 코드 가독성
- [ ] 변수/함수/클래스명이 역할을 명확히 표현한다
- [ ] 함수가 하나의 역할만 수행한다 (Single Responsibility)
- [ ] 함수의 길이가 적절하다 (기준: ~50줄 이내 권장)
- [ ] 중복 코드가 없다 (DRY 원칙)
- [ ] 매직 넘버/문자열이 상수로 정의되어 있다

### 주석 & 문서화
- [ ] 복잡한 로직에 "왜(why)"를 설명하는 주석이 있다
- [ ] Public API/함수에 적절한 문서 주석(JSDoc, docstring 등)이 있다
- [ ] 임시 코드(TODO, FIXME)에 담당자와 이슈 링크가 있다

### 테스트
- [ ] 새로운 기능에 대한 단위 테스트가 추가되었다
- [ ] 버그 수정 시 해당 케이스를 재현하는 테스트가 있다
- [ ] 테스트가 의미 있는 케이스를 커버한다 (happy path + edge case)
- [ ] 테스트가 독립적으로 실행 가능하다 (외부 의존성 없음)

### 확장성
- [ ] 설정값이 하드코딩이 아닌 환경 변수/설정 파일로 관리된다
- [ ] 향후 기능 추가 시 과도한 리팩토링 없이 확장 가능한 구조다
- [ ] 적절한 추상화 레이어가 있다 (과도한 추상화는 지양)

---

## 5. 코드 스타일 & 컨벤션

- [ ] 팀의 코딩 컨벤션/스타일 가이드를 따른다
- [ ] Linter/Formatter 규칙을 통과한다
- [ ] 커밋 메시지가 팀 컨벤션을 따른다
- [ ] 불필요한 파일(디버깅용 `console.log`, 주석 처리된 코드)이 제거되었다
- [ ] Import/의존성 정리가 되어 있다 (미사용 import 없음)

---

## 6. PR 품질

- [ ] PR 크기가 적절하다 (리뷰 가능한 범위, 기준: 400줄 이하 권장)
- [ ] PR 제목과 설명이 변경 내용을 명확히 설명한다
- [ ] 관련 이슈/티켓이 링크되어 있다
- [ ] 스크린샷/테스트 결과 등 검증 자료가 있다 (UI 변경 시)
- [ ] Breaking change가 있으면 명시되어 있다
- [ ] 마이그레이션 가이드가 있다 (필요한 경우)

## 📝 리뷰어를 위한 팁

**건설적인 피드백 작성하기**
- "이 부분을 왜 이렇게 구현하셨나요?" → "이 로직을 X 방식으로 변경하면 Y 이점이 있을 것 같아요. 혹시 현재 방식으로 하신 이유가 있을까요?"
- 코드가 아닌 행동에 집중 ("이 코드는" vs "당신은")
- 긍정적인 부분도 명시적으로 언급한다

**피드백 우선순위 표시**
- `[must]` 반드시 수정 필요
- `[suggestion]` 개선 제안 (선택사항)
- `[question]` 이해를 위한 질문
- `[nit]` 사소한 스타일 이슈
